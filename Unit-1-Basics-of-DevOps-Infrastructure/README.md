# Unit I: Basics of DevOps Infrastructure

## 1. Introduction to Containers
### Origin and Emergence of Modern Containerization
Historically, deploying applications meant running them directly on physical servers, leading to resource allocation issues. Virtual Machines (VMs) solved this by virtualizing the hardware, but VMs are heavy because each requires a full Guest Operating System. 
Containers emerged as a lightweight alternative. They virtualize the **Operating System** rather than the hardware. This means multiple containers share the host machine's OS kernel, making them incredibly fast to start and light on system resources.

### Integration into DevOps
Containers are the backbone of modern DevOps because they solve the "it works on my machine" problem. By packaging the application code along with its libraries, dependencies, and runtime environment into a single standardized unit, containers ensure consistent behavior across development, testing, and production environments.

## 2. Under the Hood: How Containers Work
Containers are not a single technology but a combination of several Linux kernel features:

### Process Isolation & Namespaces
Namespaces wrap a global system resource in an abstraction that makes it appear to the processes within the namespace that they have their own isolated instance of the global resource.
*   **PID (Process ID):** Isolates the process ID number space. A container thinks its main process is PID 1.
*   **NET (Network):** Provides isolated network interfaces, routing tables, and IP addresses.
*   **MNT (Mount):** Isolates filesystem mount points.
*   **IPC (Inter-Process Communication):** Prevents processes in different containers from communicating directly.

### Control Groups (cgroups) for Resource Limits
While namespaces isolate what a process can *see*, cgroups restrict what a process can *use*. cgroups allow administrators to allocate specific amounts of RAM, CPU time, disk I/O, and network bandwidth to a container, preventing a single container from crashing the host by consuming all available resources.

## 3. Container Images & Registries
*   **Images:** A read-only template with instructions for creating a Docker container. It contains the OS libraries, dependencies, and application code.
*   **Layers:** Images are built in layers. Each instruction in an image build creates a layer. If a layer hasn't changed, it is reused (cached) in future builds, saving time and disk space.
*   **Registries:** Services that store and distribute container images. Docker Hub is the default public registry.

## 4. Introduction to Docker Architecture
Docker is a client-server application.
*   **Docker Daemon (`dockerd`):** The server-side background process that manages Docker objects (images, containers, networks, volumes) and listens for API requests.
*   **Docker CLI (`docker`):** The client application that users interact with. It sends commands to the daemon via a REST API.
*   **Docker Registry:** The storage component where images are pulled from or pushed to.

## 5. Docker Object Types
*   **Container:** A runnable instance of an image.
*   **Image:** The blueprint for a container.
*   **Network:** Defines how containers communicate with each other and the outside world.
*   **Volume:** A mechanism for persisting data generated by a container.

## 6. Docker Layering & Filesystem
Docker uses a **Union File System (UnionFS)**. When a container is started from an image, Docker adds a read-write layer on top of the read-only image layers. This is called the **Copy-on-Write (CoW)** mechanism. If a container needs to modify a file from the read-only image, it copies that file up to the read-write layer and modifies it there, leaving the underlying image unchanged.
